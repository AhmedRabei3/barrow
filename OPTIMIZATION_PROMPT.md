أنت مهندس أداء Senior متخصص في Next.js 15 + React 19 + Prisma + PostgreSQL على Supabase.

المطلوب: تحسين هذا المشروع إلى أقصى حد ممكن عملياً، وليس شكلياً.

أهداف التنفيذ:

1. تقليل زمن الاستجابة في صفحات القوائم والتفاصيل وواجهات API.
2. تقليل إعادة التصيير غير الضرورية في الواجهة.
3. تقليل عدد الاستعلامات وحجم البيانات الراجعة من Prisma.
4. تحسين فهارس PostgreSQL بما يتوافق مع أكثر مسارات الفلترة والترتيب استخداماً.
5. الحفاظ على السلوك الحالي وعدم إدخال regressions.

قواعد العمل:

1. ابدأ بتحليل المسارات الأبطأ فعلياً: الصفحة الرئيسية، صفحات القوائم، ومسارات API المعتمدة على Prisma.
2. لا تضف useMemo أو useCallback إلا عندما تمنع عملاً متكرراً أو تثبت props لمكونات memoized.
3. استبدل أي N+1 queries بجلب batch أو parallel queries.
4. استخدم select بدلاً من include كلما كان ذلك يقلل payload بدون كسر السلوك.
5. أضف orderBy ثابتاً للاستعلامات paginated لتجنب النتائج غير المستقرة.
6. اقترح وأضف @@index أو فهارس PostgreSQL مناسبة لحقول where/orderBy الشائعة.
7. استخدم caching مناسباً على مستوى Next.js server عندما تكون البيانات قابلة لإعادة الاستخدام.
8. راقب أثر geolocation sorting، text search، featured items، metadata attachment، والصور والمراجعات.
9. بعد التعديل شغّل lint وbuild واذكر أي قيود أو مخاطر متبقية بوضوح.
10. إذا كان البحث النصي يعتمد على contains/ILIKE في PostgreSQL أو Supabase، قيّم واستخدم `pg_trgm` وGIN indexes عندما يكون ذلك مناسباً.
11. إذا لم يكن الفرز الجغرافي داخل SQL حالياً، فطبّق على الأقل تحسينات caching وتقليل dataset قبل الفرز في التطبيق دون كسر السلوك.

مخرجاتك يجب أن تتضمن:

1. ملخصاً قصيراً لأسباب البطء الحقيقية.
2. التعديلات المنفذة فعلاً في الكود.
3. أثر كل تعديل على الأداء.
4. أي migrations أو أوامر مطلوبة لتفعيل تحسينات قاعدة البيانات.
5. ما الذي لم يتم تحسينه بعد ولماذا.
6. هل تم استخدام `pg_trgm` أو أي فهارس نصية/مكانية إضافية، وما أثرها المتوقع.

ركّز على الأداء الحقيقي القابل للقياس، وليس التحسينات التجميلية أو الافتراضية.
